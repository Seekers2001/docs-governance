# docs-governance 使用说明

> 文档驱动开发插件：把项目拆成各司其职的小文档，AI 照文档干活，防止上下文"腐烂"。
> 两条主线：**① 活文档治理**（防项目文档漂移）、**② 契约式前后端协作**（防接口字段漂移）。

---

## 一、每个组件干什么

### 📘 Skill（方法论 = 说明书，AI 遇到对应场景自动参考）

| Skill | 干什么 | 什么时候触发 |
|---|---|---|
| **living-docs-governance**（活文档治理） | 把项目文档拆成**四件套**——`CLAUDE.md`(宪法/规则)、`CLAUDE_MAP.md`(地图/啥在哪)、`PROJECT_STATUS.md`(健康仪表盘/现在咋样)、`PROJECT_LOG.md`(流水账/发生过啥)，每份只干一件事、互不重复 | 项目长大、文档跟代码对不上、AI 每次进会话重新摸结构、新人接手找不到北、CLAUDE.md 越写越长 |
| **contract-first**（契约优先） | 分前后端的项目，用一份接口契约 `CONTRACT.md` 当**唯一真相源**，前后端只照它各做各的，防字段漂移（后端把 `userName` 改成 `user_name` 没通知前端 → 集成白屏） | 项目有前后端、接口字段老对不上、前端等后端联调卡住 |

### 🤖 Agent（执行者 = 照说明书干活的人）

| Agent | 干什么 |
|---|---|
| **docs-governor** | 活文档的**生成者**：扫项目、按方法论写出/更新四件套文档 |
| **docs-auditor** | 活文档的**审计员**（只读不改）：查四件套有没有漂移、重复、写了不存在的路径、指标是不是编的 |
| **contract-director** | 契约**拥有者**：定契约、改契约、最后逐字段集成对账 |
| **frontend-dev** | 前端**工人**：只读契约照着做，缺字段就上报，不自己瞎猜接口 |
| **backend-dev** | 后端**工人**：实现必须符合契约，并写"返回必须符合契约"的校验测试 |

### ⌨️ Command（入口 = 你主动敲的按钮）

| 命令 | 干什么 |
|---|---|
| **/governance** | 对当前项目做完整文档治理，生成/更新四件套 |
| **/governance-audit** | 只读审计——告诉你文档漂移在哪，不动文件 |
| **/governance-sync** | 阶段收尾时用：按"本次改动该同步哪份文档"的矩阵查漏补缺，不只追加日志 |
| **/contract** | 契约式前后端协作：先定契约，再开发，最后对账 |

## 二、关系图

```
文档治理线：  /governance ──→ docs-governor（写四件套）
              /governance-audit ──→ docs-auditor（查漂移）
              /governance-sync ──→ docs-governor（按矩阵收尾同步）
              └─ 方法论来自 living-docs-governance skill

前后端协作线：/contract ──→ contract-director（契约拥有者）
                            ├─ frontend-dev（前端工人）
                            └─ backend-dev（后端工人）
              └─ 方法论来自 contract-first skill
```

最常用就两个：平时维护项目敲 `/governance`，分前后端开发敲 `/contract`。

---

## 三、两种协作模式（契约线）

契约式协作有两种用法，按你的实际场景选：

1. **单会话多 agent**：一个 Claude 会话里，`contract-director` 派出 `frontend-dev` + `backend-dev` 并行开发。适合一个人一个会话内推进。
2. **双终端各自跑**（推荐给习惯双终端的）：终端1 跑前端、终端2 跑后端，**两边各自独立**，没有实时调度——**协调全靠那份 `CONTRACT.md` 文件**（异步媒介）。这时"主任"退化成"契约的拥有者/编辑者"，谁改契约谁就是主任。

不管哪种模式，铁律一致：**接口只在 `CONTRACT.md` 定义一次，谁都不许在实现里私自偏离；要改接口先改契约。**

## 四、专业术语对照（写简历/对外介绍时用）

| 通俗说法 | 学名 |
|---|---|
| 前后端各做各的、只通过 API 沟通的架构 | **前后端分离**（decoupled frontend-backend） |
| 两边写完合到一起调试 | **联调**（joint debugging / integration） |
| 本插件契约线这套方法论 | **契约测试 / 消费者驱动契约（Consumer-Driven Contracts, CDC）** |
| "消费方需求先行"（先看前端要什么再定接口） | 正是 **CDC 的核心原则** |
| 后端写"返回必须符合契约"的测试 | **提供者验证**（provider verification） |
| 契约文档本身 | **API 契约 / 接口规格**（OpenAPI / Swagger） |
| 标杆工具 | **Pact**（契约测试框架） |

> 一句话：本插件契约线 ≈ 一套轻量的 **Consumer-Driven Contract testing** 实践，用一份 Markdown 契约 + agent 角色分工落地。

## 六、安装位置

已通过符号链接装进 `~/.claude/`（skills / agents / commands），指向本目录，**改本目录源文件即自动同步**。hooks 默认未装（`hooks/check-on-stop.sh` 是 Stop 钩子，按需手动启用）。
